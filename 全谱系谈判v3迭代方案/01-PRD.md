# 全谱系博弈训练系统 v3 — 产品需求文档（PRD）

> 文档版本：1.0 ｜ 状态：待确认 ｜ 迭代起点：v2.0（单文件 `game_trainer.html`，83KB / 1535行）

---

## 1. 背景与问题（Why）

v2.0 已实现一个可在浏览器离线运行的单文件谈判训练系统，包含 7 大博弈场景、6 种 AI 对手、六大宗师理论库、8 维心理画像与 LocalStorage 持久化。它**功能完整但存在结构性瓶颈**：

| 痛点 | v2.0 现状 | 影响 |
|------|-----------|------|
| **单文件巨石** | 1535 行 HTML/CSS/JS 全部塞在一个文件 | 难维护、难协作、难测试、加一个场景要全文件搜索 |
| **AI 对手"假"** | 决策基本是 `Math.random() < 阈值`，跨回合无记忆 | 6 个对手玩起来差异感弱，无法形成"识破对手→针对性应对"的训练闭环 |
| **模块强耦合** | UI 渲染、业务逻辑、数据混在全局函数里，靠全局 `state` 变量串联 | 改 UI 会碰逻辑，改逻辑会碰数据，回归风险高 |
| **无成长反馈深度** | 心理画像每局固定 `+0.08` 线性累加，与实际决策弱相关 | 画像失真，训练指导意义有限 |
| **复盘浅** | 仅展示最终比分 + 固定话术 | 用户学不到"我哪一步该怎么改" |

**v3 的使命**：在不牺牲"单文件可离线分发"这一核心优势的前提下，把系统重构为**模块化、事件驱动、AI 真正可博弈**的训练平台，并显著提升对手智能与复盘深度。

---

## 2. 产品目标（What）

### 2.1 北极星指标
**单局训练的"学习获得感"** —— 用户每打完一局，都能清楚知道：对手是什么类型、我哪几步打得好/差、下次该用什么策略。

### 2.2 v3 三大支柱

1. **架构模块化**：拆分为 6 个 JS 模块目录 + 事件总线，模块间零直接调用，全部通过事件通信。开发态多文件，分发态可一键打包回单文件 HTML。
2. **AI 对手智能化**：每个对手拥有差异化决策引擎 + 跨回合记忆（记住玩家历史行为并调整策略），形成"性格 → 行为模式 → 可被识破的破绽"的完整博弈体验。
3. **复盘深度化**：逐回合决策评分、关键转折点标注、对手心理状态追踪、针对性改进建议。

### 2.3 明确不做（Non-Goals）
- ❌ 不引入后端/账号系统（保持纯前端离线）
- ❌ 不引入构建框架（React/Vue）—— 保持零运行时依赖，用原生 ES Module
- ❌ 不做联机对战
- ❌ v3 不重写理论库内容（六大宗师/策略卡数据沿用 v2.0，仅迁移）

## 3. 功能需求

### 3.1 保留并迁移（v2.0 → v3 平移）
- 7 大场景：囚徒困境、最后通牒、信任博弈、商业谈判、危机谈判、公共品博弈、联盟谈判
- 6 大宗师理论库 + 策略卡 + 沟通技术 + 心理防御
- 玩家档案 / 军衔 / LocalStorage 持久化
- 赛博朋克终端 UI 风格

### 3.2 新增 / 增强（v3 核心增量）

| 编号 | 需求 | 优先级 | 验收要点 |
|------|------|--------|----------|
| F-01 | **事件总线**：所有模块通过 `EventBus.emit/on` 通信 | P0 | UI 不直接调用 engine，engine 不直接操作 DOM |
| F-02 | **AI 决策引擎**：6 对手独立策略函数 + 统一接口 | P0 | 同一场景同一玩家操作，6 对手产生可观测的不同应对 |
| F-03 | **跨回合记忆**：对手记录玩家历史动作（合作率/激进度/让步幅度），并据此调整 | P0 | 玩家连续背叛后，"理性型/合作型"对手会转向报复 |
| F-04 | **对手情绪/信任状态机**：每个对手维护 `trust/anger/patience` 等内部状态，随回合演化 | P1 | 复盘可展示对手情绪曲线 |
| F-05 | **逐回合复盘评分**：每个决策给出 0-100 评分 + 简评 | P1 | 危机谈判每阶段显示"这步评分及更优解" |
| F-06 | **心理画像引擎重构**：基于真实决策序列计算 8 维 | P1 | 画像值与玩家实际行为强相关，非固定累加 |
| F-07 | **打包脚本**：`build.js` 将多模块合并为单文件 HTML | P0 | `node build.js` 产出 `dist/game_trainer.html` 可双击运行 |
| F-08 | **对手"破绽提示"系统**：训练中可消耗提示，揭示对手当前心理状态 | P2 | 与 v2.0 的 `PERSONALITY_TELLS` 联动 |

## 4. 用户故事

- **作为谈判学习者**，我想在打囚徒困境时，连续背叛后看到"陈逻辑"开始报复我，这样我能体会"重复博弈中声誉的代价"。
- **作为谈判学习者**，我想在一局结束后看到"你第3轮的激进出价是本局最大失误，对手是风险规避型，此时应小步施压"，这样我能精准改进。
- **作为开发/维护者**，我想新增一个场景时只需在 `scenarios/` 加一个文件并注册，不必改动其他模块。

## 5. 验收标准（v3 整体 Done 定义）

1. 开发态：`src/` 下 6 个模块目录，模块间仅通过事件总线 + 显式 import 数据通信。
2. 分发态：`node build.js` 一键产出单文件 `dist/game_trainer.html`，双击浏览器运行，功能与体验 ≥ v2.0。
3. AI：6 对手在同一场景下行为可区分；至少在囚徒困境、商业谈判、信任博弈三个场景实现跨回合记忆。
4. 复盘：至少 3 个场景提供逐回合评分。
5. 回归：v2.0 全部功能在 v3 中可用，无功能回退。
6. 数据兼容：v2.0 的 LocalStorage 玩家存档在 v3 可读取（或提供迁移）。

## 6. 风险与权衡

| 风险 | 说明 | 对策 |
|------|------|------|
| ES Module 的 `file://` 限制 | 浏览器对本地 `file://` 直接打开的 ES Module 有 CORS 限制 | 开发态用本地 http server；**分发态用 build.js 打包成无 import 的单文件**，规避限制 |
| 重构引入回归 | 巨石拆分易漏功能 | Phase 化迁移 + 每 Phase 对照 v2.0 验收 |
| AI 过度复杂 | 决策引擎可能过度工程化 | 先用"状态机 + 加权规则"，不上机器学习 |
| 单文件优势丢失 | 用户核心诉求是"复制粘贴就能用" | build.js 必须作为 P0 与架构同步交付 |
