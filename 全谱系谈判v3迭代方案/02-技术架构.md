# 全谱系博弈训练系统 v3 — 技术架构

> 配套 `01-PRD.md`。本文定义 6 个 JS 模块目录、事件总线契约、数据流与打包方案。

---

## 1. 架构原则

1. **零运行时依赖**：原生 ES Module，不用框架、不用 npm 包（构建脚本 `build.js` 用 Node 仅在开发期运行）。
2. **事件驱动解耦**：模块之间**不允许互相直接 import 行为函数**；通信一律走 `EventBus`。数据常量（如对手定义、宗师理论）允许被 import。
3. **单向数据流**：`UI 操作 → emit 事件 → engine/scenario 处理 → 更新 store → emit 状态变更 → UI 重渲染`。
4. **可打包回单文件**：开发态多文件，`build.js` 内联所有模块产出单一 `dist/game_trainer.html`。

## 3. 事件总线契约

### 3.1 EventBus API
```js
EventBus.on(event, handler)     // 订阅，返回取消函数
EventBus.once(event, handler)   // 一次性订阅
EventBus.off(event, handler)    // 退订
EventBus.emit(event, payload)   // 派发
```

### 3.2 核心事件清单（`core/events.js`）

| 事件名 | 发出方 | 监听方 | payload |
|--------|--------|--------|---------|
| `nav:goto` | UI | router | `{ screen, params }` |
| `game:start` | UI(场景选择) | scenario | `{ scenarioKey, opponentId }` |
| `game:render` | scenario | ui/game-view | `{ html / viewModel }` |
| `player:action` | UI | scenario | `{ type, value }` |
| `ai:decided` | engine | scenario | `{ move, mood, reason }` |
| `round:end` | scenario | analytics | `{ roundLog }` |
| `game:end` | scenario | analytics/ui | `{ result }` |
| `analytics:scored` | analytics | ui/result | `{ replay, score }` |
| `profile:updated` | analytics | store/ui | `{ profile }` |
| `store:changed` | store | ui | `{ key, value }` |

### 3.3 关键交互序列（以一回合为例）

```
玩家点击"背叛"
  └─ UI emit player:action {type:'choice', value:'defect'}
       └─ scenario 接收 → 调 engine.decide(opponentId, 'prisoners', ctx)
            └─ engine 查 memory + mood → 产出 move → emit ai:decided
                 └─ scenario 结算本回合 → emit round:end → emit game:render(下一回合)
                      └─ ui/game-view 重渲染
```

## 4. 模块依赖规则（强约束）

```
ui ──emit/on──▶ core(EventBus/store)
scenarios ──emit/on──▶ core，并 import engine + data
engine ──import──▶ data + memory/mood，emit ai:decided
analytics ──on──▶ round:end/game:end，emit analytics:scored
data ── 纯常量，被任何模块 import，自身不依赖任何模块
```

- ✅ 允许：任何模块 `import` `data/` 与 `core/event-bus.js`
- ✅ 允许：`scenarios` `import` `engine`
- ❌ 禁止：`ui` 直接 `import` `engine` 或 `scenarios` 的逻辑函数
- ❌ 禁止：`engine` 操作 DOM

## 5. 打包方案（build.js）

**目标**：把 `src/` 的 ES Module 图谱 + CSS + HTML 内联为单个 `dist/game_trainer.html`。

**策略（简单可靠，不引第三方打包器）**：
1. 读取 `src/index.html`，定位 `<script type="module">` 入口。
2. 从入口 `bootstrap.js` 出发，递归解析 `import` 语句，按拓扑序收集所有模块源码。
3. 剥离每个模块的 `import`/`export` 关键字，按依赖顺序拼接进一个 IIFE 作用域（或挂载到一个内部命名空间对象）。
4. 内联 `styles.css` 进 `<style>`，拼接后的 JS 进 `<script>`（非 module）。
5. 输出 `dist/game_trainer.html`。

> 备选：若手写依赖解析风险高，Phase 1 可先用「约定顺序的手工 manifest 列表」拼接（在 build.js 里显式列出文件顺序），后续再升级为自动拓扑解析。**Phase 1 采用 manifest 顺序拼接，确保尽早可分发。**
